# Golden T14 — adversarial verify before trusting a root cause (E4, tabletop)

**Registered:** 2026-07-02, before E4 implementation.

## Scenario setup
An `/iterate` run is debugging intermittent 502s on an API. In Phase 1 the model
forms ONE hypothesis ("the upstream timeout is too low") after reading one log
line, and the bug's shape admits several plausible causes (timeout, connection
pool exhaustion, OOM restarts). The model is about to apply the timeout fix on
the strength of that single unchallenged hypothesis.

## Expected transcript
1. Because the fix is about to be trusted on a root-cause claim that has not been
   challenged (multi-cause bug, single hypothesis, thin evidence), the
   adversarial-verify option fires: spawn a skeptic subagent explicitly prompted
   to REFUTE the hypothesis (argue for rival causes, point at disconfirming
   evidence).
2. The fix is NOT applied while the hypothesis stands unrefuted-but-unchallenged.
   Either the skeptic fails to refute it (hypothesis survives a real challenge →
   proceed to fix), or the skeptic surfaces a rival cause → new targeted check
   before any fix.
3. The DIAGNOSIS block records that the hypothesis was adversarially challenged
   and the outcome.

## Pass rule (binding, from epics doc)
Adversarial verify: skeptic attempts refutation before fix is trusted; PASS iff
fix isn't applied on an unrefuted-but-unchallenged hypothesis.
